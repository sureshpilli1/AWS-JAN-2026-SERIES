AWS Identity and Access Management (IAM) Policies – Complete Documentation

1. What is an IAM Policy?
An **IAM Policy** is a document that defines **permissions** in an AWS environment.
In simple terms:
* A policy answers **who can do what on which AWS resource**
* Policies are written in **JSON format**
* Policies are attached to:
  * Users
  * Groups
  * Roles
  * AWS resources (in some cases)
IAM policies follow a **permission-based access control model**.

2. Why IAM Policies Are Important
Without IAM policies:
* Everyone would have full access
* Security risks increase
* Accidental deletion of resources can happen
* Compliance becomes impossible
IAM policies help enforce:
* Least privilege
* Separation of duties
* Controlled access
* Auditable permissions

3. Types of IAM Policies
AWS supports **three major policy types**.

4. AWS Managed Policies
What are AWS Managed Policies?
These are **predefined policies created and maintained by AWS**.
Characteristics:
* Created by AWS
* Automatically updated when services change
* Cannot be edited
* Cannot be deleted by customers
Examples
* `AmazonS3ReadOnlyAccess`
* `AmazonEC2FullAccess`
* `ReadOnlyAccess`
When to Use
* Standard access requirements
* Quick setup
* Less customization needed

5. AWS Managed Policies – Job Function Policies
What are Job Function Policies?
These are **AWS-managed policies designed around common job roles**.
AWS maps typical job responsibilities to service-level permissions.
Examples
DBA (Database Administrator)
* Database services → Full access
* Storage services → Limited access
* Compute services → Read-only access
Network Administrator
* VPC & networking services → Full access
* Compute services → Read-only access
Key Points
* Cannot be edited or deleted
* Useful for role-based access
* Best for initial role setup
* Often too broad for strict security environments

6.Customer Managed Policies
What are Customer Managed Policies?
These are **custom policies created by customers** based on their specific requirements.
Characteristics:
* Fully editable
* Can be deleted
* Version controlled by AWS
* Reusable across users, groups, and roles
AWS stores **multiple versions**, but only **one version is active** at a time.
When to Use
* Fine-grained permissions
* Production environments
* Compliance-driven access control
* Organization-specific requirements

7. IAM Policy Structure (Core Elements)
Every IAM policy contains the following key components.
7.1 Service (Action Namespace)
This defines **which AWS service the policy applies to**.
Examples:
* `s3` → Amazon S3
* `ec2` → Amazon EC2
* `rds` → Amazon RDS
Multiple services can be included in the same policy.
7.2 Effect
Defines **whether the action is allowed or denied**.
Possible values:
* `Allow`
* `Deny`
Important rule:
* **Explicit Deny always overrides Allow**
7.3 Action
Specifies **what operations are permitted or denied**.
Examples:
* S3:
  * `s3:GetObject`
  * `s3:PutObject`
  * `s3:DeleteObject`
* EC2:
  * `ec2:StartInstances`
  * `ec2:StopInstances`
  * `ec2:DescribeInstances`
Wildcard (`*`) can be used to allow all actions for a service.
7.4 Resource
Defines **which AWS resources the policy applies to**.
Examples:
* Specific S3 bucket
* Specific EC2 instance
* All resources using `*`
Using `*` means:
* Permission applies to **all resources of that service**

8. Policy Evaluation Logic (Very Important)
When a request is made, AWS evaluates permissions in this order:
1. Explicit Deny
2. Explicit Allow
3. Default Deny (if nothing matches)
If no policy allows the action → **access is denied by default**.

9. Requirement Scenario (Custom Policy)
Requirement
Create a **custom IAM policy** that allows a user to access:
* Amazon S3
* Amazon EC2
Access Level
* Allow actions on both services
* No restriction on specific resources
This requirement is typically used for:
* Training accounts
* Lab environments
* Limited admin users

10. Custom IAM Policy – S3 + EC2 Access
Sample Customer Managed Policy (JSON)

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3Access",
      "Effect": "Allow",
      "Action": "s3:*",
      "Resource": "*"
    },
    {
      "Sid": "AllowEC2Access",
      "Effect": "Allow",
      "Action": "ec2:*",
      "Resource": "*"
    }
  ]
}
```

---

11. Explanation of the Policy
Service
* Targets **Amazon S3** and **Amazon EC2**
Effect
* `Allow` → Permissions are granted
Action
* `s3:*` → All S3 operations
* `ec2:*` → All EC2 operations
Resource
* `*` → Applies to all buckets and instances

12. Important Notes (Teaching & Production)
* This policy gives **broad access**
* Not recommended for production users
* Useful for:

  * Labs
  * Training
  * Temporary admin access

For production:
* Restrict actions
* Restrict resources
* Use conditions
* Prefer roles over users

13. Comparison Summary

| Policy Type      | Editable | AWS Maintained | Best Use Case        |
| ---------------- | -------- | -------------- | -------------------- |
| AWS Managed      | ❌        | ✅              | Standard access      |
| Job Function     | ❌        | ✅              | Role-based access    |
| Customer Managed | ✅        | ❌              | Fine-grained control |

14. Best Practices
* Avoid using `*` in production
* Use customer managed policies for control
* Attach policies to groups, not users
* Review policy versions periodically
* Test policies using IAM Policy Simulator
* Always follow least privilege

15. Final Summary
IAM policies are the **foundation of AWS security**.
Understanding:
* Policy types
* JSON structure
* Evaluation logic
* Custom policy creation

is mandatory for:

* AWS engineers
* DevOps roles
* Cloud security
* Real-world production environments





1. Overall meaning
This policy says:
Allow certain S3 actions on a specific bucket (my-company-logs)

2. Top-level fields
{
  "Version": "2012-10-17",
  "Statement": [ ... ]
}
Version
"2012-10-17"
This is the policy language version, not a date you change
Always use this value for modern policies
Statement
Statement = a permission rule
It is an array because a policy can have multiple rules
Each object inside the array = one rule
Policy = Collection of rules
Statement = One rule
This is the main rule block
You can have multiple statements
Each statement = one permission rule

3. Inside Statement (core part)

{
  "Sid": "AllowS3AccessToSpecificBucket",
  "Effect": "Allow",
  "Action": [...],
  "Resource": [...]
}

4. Sid (Statement ID)
"Sid": "AllowS3AccessToSpecificBucket"
Just a name/label for the rule
Used for:
Identification
Readability
Sid = "Statement ID"
Optional (can be removed, but good practice to keep)

5. Effect
"Effect": "Allow"
Defines:
Allow OR Deny
"Allow" → grant permission ✔
"Deny" → explicitly block ❌

6. Action
"Action": [
  "s3:GetObject",
  "s3:PutObject",
  "s3:ListBucket"
]
These are operations allowed
Meaning of each:
s3:GetObject
Read/download files from S3
s3:PutObject
Upload files to S3
s3:ListBucket
List files inside bucket

7. Resource (VERY IMPORTANT)
"Resource": [
  "arn:aws:s3:::my-company-logs",
  "arn:aws:s3:::my-company-logs/*"
]
Defines:
Which resources these permissions apply to

8. What is ARN?
ARN = Amazon Resource Name
It uniquely identifies AWS resources.
General format
arn:partition:service:region:account-id:resource

9. Your ARN explained
Bucket ARN
arn:aws:s3:::my-company-logs
Breakdown:
arn → standard prefix
aws → partition
s3 → service
::: → S3 doesn’t use region/account here
my-company-logs → bucket name
This refers to:
THE BUCKET itself
Object ARN
arn:aws:s3:::my-company-logs/*
Meaning:
ALL objects inside the bucket

10. Why two ARNs?
Because different actions apply to different levels
Example:
| Action     | Requires   |
| ---------- | ---------- |
| ListBucket | Bucket ARN |
| GetObject  | Object ARN |
| PutObject  | Object ARN |

So you need BOTH:
"arn:aws:s3:::my-company-logs"       → for ListBucket
"arn:aws:s3:::my-company-logs/*"     → for objects

11. Full meaning of your policy
In simple words:
Allow:
- listing files in the bucket
- uploading files
- downloading files
ONLY for:
- bucket: my-company-logs
- objects inside it

12. Real-world example
If this policy is attached to:
EC2 instance
Kubernetes pod (via IAM role)
Lambda
That entity can:
✔ Upload logs
✔ Read logs
✔ List logs
Cannot access other buckets

13. Important concepts
Least privilege
This policy follows:
Give only required permissions ✔
Not global access
It does NOT allow:
arn:aws:s3:::*
So it’s restricted ✔

14. Common mistakes
Missing /* → cannot access objects
Only object ARN → cannot list bucket
Wrong bucket name
Using "*" (too broad, insecure)

15. One-line memory
ARN = unique identifier of AWS resource
Final summary
This IAM policy:
Allows S3 read/write/list access
Only for my-company-logs bucket
Uses ARN to define resource scope
Uses Sid for identification

------------------------

As a DevOps engineer, you’ll see JSON everywhere:
AWS IAM policies ✔

CI/CD tools ✔

You must be able to read and write JSON confidently

2. What you actually need to learn (not everything)
You don’t need deep theory. Just learn these basics:
1. Structure
{
  "key": "value"
}
{} → object
[] → list/array

"name": "suresh"
3. Arrays
"actions": ["s3:GetObject", "s3:PutObject"]
4. Nesting
{
  "Statement": [

      "Effect": "Allow"

  ]
}

Strings must be in double quotes
No trailing commas

Complex parsing
Writing large JSON from scratch

🔹 4. Real DevOps approach (practical truth)
In real life, you rarely write JSON from scratch.
You will:
Copy from docs

what is nesting and how to write it 
-----------------------------------
What is nesting?
Nesting means putting one structure inside another
In JSON:
A value can itself be another object {} or array []
1. Simple example (no nesting)
{
  "name": "suresh",
  "age": 25
}
Flat structure (no nesting)

2. Nested object
{
  "name": "suresh",
  "address": {
    "city": "Hyderabad",
    "state": "Telangana"
  }
}
Here:
"address" → contains another object
This is nesting

3. Nested array
{
  "skills": ["docker", "kubernetes", "aws"]
}
Array inside object = nesting

4. Deep nesting (multiple levels)
{
  "user": {
    "name": "suresh",
    "projects": [
      {
        "name": "k8s-app",
        "status": "running"
      }
    ]
  }
}

Structure:
user
 ├── name
 └── projects
       └── object inside array

5. Real DevOps example (IAM style)
{
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject"],
      "Resource": "*"
    }
  ]
}
Nesting levels:
Statement → array
   → object
      → Action → array
6. Why nesting is used
To organize complex data
Instead of this:
{
  "user_name": "suresh",
  "user_city": "Hyderabad"
}
We use nesting :

{
  "user": {
    "name": "suresh",
    "city": "Hyderabad"
  }
}

7. Real-world analogy
Think of:
Folder → Subfolder → Files
Example:
Company
 └── Employee
      └── Address
That’s nesting

8. Simple memory trick
Nesting = object inside object OR array inside object

9. Common mistakes
Missing brackets
Wrong nesting level
Confusing array vs object
🔚 Final summary
Nesting means:
Putting structured data inside another structure
Used to:
Organize data
Represent real-world relationships
